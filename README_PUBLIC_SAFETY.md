# Public Safety System

This system integrates `mafiarp_police_system`, `detective`, and
`realistic_police` behind one server-authoritative service.

## Ownership

- Police: CAD, profiles, warrants, BOLOs, arrests, vehicles, firearms, garage,
  radio, tickets, notices, and police NPC services.
- Detective: cases, scenes, fingerprints, evidence, comparisons, photos, and
  warrant requests.
- Realistic Police: surrender, cuffs, escort, transport, search, booking,
  incarceration, bail, CCTV, trunk equipment, and terminal intrusion.

All cross-plugin calls use `ix.publicSafety` and registered adapters. Optional
rank, clothing, vehicle, property, and banking integrations may be absent; the
corresponding action is denied or hidden instead of raising an error.

## Configuration

New-system configuration is centralized in `mafiarp_police_system/sh_plugin.lua`
under `PLUGIN.PublicSafetyConfig`.

- `policeFactions`: exact Helix faction names treated as police.
- `requireDuty`: requires the existing police duty state.
- `permissions`: minimum rank and optional flag for every protected operation.
- `stationRange` / `fieldRange`: physical interaction limits.
- `requestsPerSecond`: fallback request-rate ceiling.
- `searchPageSize`: SQL result page size, clamped to 1-50.
- `maxOpenWindows`: CAD window cap.
- `vehicleComputerKey`: reduced vehicle-CAD bind.
- `vehicleClasses`: classes allowed to provide vehicle CAD.
- `custody`: action times, tools, seats, booking, property, charges, and bail.
- `warrants`: server-derived warrant and BOLO lifetimes.
- `tickets`: due time, amount cap, and open-ticket cap.
- `detective`: scene/evidence/photo limits and supply behavior.
- `camera`: range, viewer cap, sabotage, repair, and intrusion timing.
- `intrusion`: physical tool requirements and session lifetime.
- `trunk`: placement limits, global/officer caps, and per-model stock.
- `retention`: audit and released-scene retention.

The optional `trunk.spikeStrip.model` is blank by default because the installed
content contains no spike-strip model. Set it to a valid mounted model path to
enable it; invalid or missing models remain hidden cleanly.

## CAD Access

Use a `realistic_police_computer` for the full station desktop. The server
checks the entity, range, faction, duty, and application permissions throughout
the session.

The reduced vehicle CAD opens with `vehicleComputerKey` while seated in an
allowed police vehicle. It intentionally excludes CCTV, editing, approvals,
firearm administration, garage administration, and department administration.

The CAD uses typed, paginated requests and detail-on-selection. It never sends
the complete character database. Window positions and recent searches are
session-only.

## Operational Flow

1. A detective opens the casework desk, creates a case and scene, then collects
   prints, evidence, and photographs.
2. Booking creates the authoritative arrest/booking record and a known-print
   reference.
3. A detective submits a warrant request from the case.
4. An authorized station-CAD user approves, rejects, or requests changes.
5. Field officers use the context menu for identity, pat-down, restraints,
   escort, authorized search, booking handoff, plate lookup, citations, and
   impound marking.
6. The jailer performs booking and property storage. The bailer validates the
   exact prisoner, booking, amount, and payer.
7. Release or bail returns stored property and updates linked systems.

## Persistence And Migration

Normalized data uses additive `ix_ps_*` SQL tables through Helix's database
wrapper, so both SQLite and mysqloo are supported.

The `public_safety_legacy_v1` migration imports:

- `jr_police_cases_database`
- `jr_police_weapon_database`
- `data/realistic_police/record/*.txt`
- `data/realistic_police/report/*.txt`

Every imported row is checked by `legacy_source` before insertion. The final
migration marker is written only after asynchronous work completes. Existing
source data is not deleted.

Fingerprint surfaces, samples, and reference cards remain in their normalized
detective tables and are restored as persisted records after reconnect.

## Security Model

- Immutable record IDs are generated by the server.
- Monetary values, sentences, ownership, item instances, entities, ranges,
  permissions, custody transitions, and timestamps are validated server-side.
- Sensitive operations use per-record locks or serialized queues.
- CAD, detective desks, booking, bail, disposal, and intrusion use expiring
  physical sessions.
- Sensitive changes are written to `ix_ps_audit` and Helix logs.
- CCTV only adds the actively viewed camera to a viewer's PVS.
- Evidence transfers are ordered and every custody/seal change is logged.

## Intentionally Retired Duplicates

The server denies the old bulk character, old notes, old case editing, old
report/criminal-record, old camera-control, old arrest/bail, and old terminal
hacking network workflows. Their network names remain registered so stale
clients fail safely.

The authoritative replacements are the station/vehicle CAD, detective
casework, public-safety booking/bail, CCTV app, and restricted intrusion panel.
The existing ticket, jail-bed contraband, finebook, plate rendering, trunk
preview, camera world entities, and physical custody presentation are retained.

## Maintenance

After changing permissions, factions, vehicle classes, charge definitions, or
trunk stock, restart the server so shared client/server configuration agrees.
Do not rename existing `ix_ps_*` tables or record prefixes.

Run the local checks with:

```powershell
$env:NODE_PATH="C:\path\to\node_modules"
node .\tests\glua_syntax_test.js
node .\tests\public_safety_contract_test.js
```

`glua_syntax_test.js` requires the `luaparse` Node package.
